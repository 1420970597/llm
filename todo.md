# TO C 竞品级改版方案 / TODO

## 0. 改版总目标

把当前系统从“数据工厂后台”重构为**普通用户也能顺畅上手、愿意持续使用、信任感足够强**的产品。

一句话目标：

> 用户不需要理解模型、存储、图谱、奖励数据、JSONL 工件这些内部概念，也能完成“创建任务 → 等待生成 → 查看结果 → 继续优化/导出”的完整闭环。

---

## 1. 核心判断

当前系统最大问题不是 UI 不够漂亮，而是：

1. **信息架构错了**：现在按内部流水线组织，不是按用户目标组织。
2. **起步成本太高**：用户一上来就被迫做系统级决策。
3. **异步过程黑盒**：只会“已入队”，没有真正的任务反馈。
4. **结果不可判断**：更像数据库浏览器，不像成品结果页。
5. **产品信任感不够**：默认账号、后台术语、研发信息泄露都在伤害 TO C 感知。

---

## 2. 新产品定位

### 2.1 面向谁
面向非技术或弱技术用户，他们的目标不是“操作数据流水线”，而是：

- 输入一个主题/方向
- 让系统自动生产内容或数据
- 在少量关键节点做确认
- 获得可理解、可筛选、可导出的结果

### 2.2 用户心智
系统不再强调：
- 领域图谱
- 奖励数据
- 工件
- Provider
- Storage Profile

系统改为强调：
- 我想做什么
- 现在进行到哪一步
- 需要我做什么确认
- 结果质量怎么样
- 下一步建议是什么

### 2.3 产品主叙事
新的主叙事应该是：

1. 创建一个任务
2. 系统自动拆解和生成
3. 你只在关键点确认
4. 最终获得可用结果

而不是：

1. 配策略
2. 选模型
3. 选存储
4. 审图谱
5. 再点各阶段生成
6. 再看导出工件

## 3. 新信息架构（直接替换现有导航）

## 3.1 旧导航问题
当前导航：
- 总览
- 计划编排
- 领域图谱
- 问题生成
- 推理生成
- 奖励数据
- 导出交付

问题：这套导航是**给系统实现看的，不是给用户看的**。

## 3.2 新导航方案
建议统一改为以下 5 个一级入口：

1. **首页**
2. **我的任务**
3. **结果中心**
4. **模板与高级设置**
5. **账户与帮助**

管理员额外看到：
6. **系统管理**

## 3.3 每个入口的职责

### 首页
回答三个问题：
- 我现在能做什么？
- 我最近的任务进展如何？
- 我下一步最应该点哪里？

### 我的任务
任务制替代流水线制。
用户看到的是：
- 任务名
- 状态
- 当前阶段
- 预计剩余时间
- 最近一次更新
- 是否需要我确认

### 结果中心
用户看到的是：
- 已完成结果
- 质量评分
- 推荐查看项
- 异常/低质量项
- 导出入口

### 模板与高级设置
默认隐藏。
仅对高级用户开放：
- 生成风格
- 深度/速度偏好
- 输出格式
- 可选高级参数

### 账户与帮助
包括：
- 账号信息
- 新手帮助
- FAQ
- 联系支持
- 产品术语说明

### 系统管理（管理员）
保留现在的治理能力，但必须和普通用户区隔。

## 4. 新主流程（替换现有用户链路）

## 4.1 新用户主路径

### Step 1：创建任务
用户只填写最少字段：
- 任务名称
- 主题/关键词
- 目标规模（简单档位）
- 输出类型（默认即可）

默认不让普通用户一上来决定：
- provider
- storage
- strategy
- prompt
- reasoning effort

### Step 2：系统自动规划
系统自动完成：
- 规模估算
- 结构拆分
- 默认模型选择
- 默认存储落点
- 默认策略匹配

给用户看的不是内部参数，而是：
- 预计耗时
- 预计输出量
- 是否建议继续

### Step 3：关键节点确认
用户只在真正需要判断的地方出现：
- 方向结构是否合理
- 是否开始正式生成
- 是否接受当前结果

### Step 4：生成中
用户看到：
- 当前阶段
- 总进度
- 每阶段进度
- 异常提醒
- 可否离开页面
- 完成后通知

### Step 5：查看结果
用户看到：
- 高质量结果摘要
- 低质量/失败项
- 推荐下一步
- 一键导出

## 5. 首页重构方案

首页必须从“统计看板”改成“任务指挥页”。

## 5.1 首页应包含

### 模块 A：欢迎区
- 你今天想生成什么？
- 一个明显的“新建任务”主按钮
- 最近一次任务状态提示

### 模块 B：继续进行中的任务
- 当前阶段
- 百分比进度
- 是否卡住
- 一键继续

### 模块 C：最近结果
- 最近完成的任务
- 质量概览
- 导出入口

### 模块 D：新手引导
首次进入时展示：
- 3 步开始
- 示例任务
- 常见问题

## 5.2 首页必须删除的内容
- 后台式总数统计优先级过高
- “统一覆盖用户链路与管理员治理”这种研发叙事
- 系统迁移背景
- 仓库路径/提交时间

---

## 6. 创建任务页重构方案

## 6.1 当前问题
当前创建路径要求用户先理解：
- 生成策略
- 模型提供方
- 存储配置
- 估算机制

这是典型的后台配置思路。

## 6.2 新建任务页应该怎么做

### 普通模式（默认）
只展示：
- 任务名称
- 主题关键词
- 目标规模（小 / 中 / 大）
- 输出倾向（更快 / 更平衡 / 更深入）

### 高级模式（折叠）
点击“高级设置”才展开：
- 自定义模板
- 更细粒度规模
- 语言/风格偏好
- 导出格式偏好

### 提交前必须展示
- 预计耗时
- 预计产出规模
- 是否消耗额度
- 是否可以稍后继续

## 6.3 新交互原则
- **先让用户开始，再逐步暴露复杂度**
- 不允许“必须先估算才能创建”这种硬阻塞体验
- 估算结果应该内嵌展示，而不是单独变成前置关卡

---

## 7. 任务系统重构方案

## 7.1 核心原则
“已入队”不能算用户反馈。

必须引入真正的任务系统。

## 7.2 每个任务必须有
- `task_id`
- `type`
- `status`
- `current_stage`
- `progress_percent`
- `started_at`
- `updated_at`
- `estimated_finish_at`
- `retryable`
- `error_summary`
- `next_action`

## 7.3 状态机
统一使用：
- `draft`
- `planning`
- `waiting_confirmation`
- `queued`
- `running`
- `partial_failed`
- `failed`
- `completed`
- `export_ready`
- `cancelled`

## 7.4 UI 呈现要求
任务卡必须清楚展示：
- 当前在做什么
- 还剩多少
- 如果失败了要怎么办
- 下一步按钮是什么

例如：
- 正在生成结构（42%）
- 等待你确认方向
- 生成失败，可重试
- 已完成，可查看结果

---

## 8. 结果中心重构方案

## 8.1 当前问题
现在结果页像记录浏览器：
- objectKey
- artifactType
- 原始列表
- 内部字段

这对普通用户几乎没有价值。

## 8.2 新结果页应该包含

### 结果摘要卡
- 本次任务产出总量
- 推荐结果数
- 异常项数
- 质量概览

### 结果浏览区
用户看到的是：
- 内容预览
- 推荐理由
- 质量标签
- 是否可导出

### 风险区
- 哪些内容可能质量较低
- 哪些步骤失败过
- 哪些结果建议重新生成

### 导出区
只讲用户能理解的话：
- 下载结果
- 下载全部
- 下载筛选结果

不要默认暴露：
- objectKey
- bucket 路径
- artifactType

这些只应出现在管理员/高级调试视图。

---

## 9. 领域图谱页重构方案

## 9.1 当前问题
“领域图谱”这个概念本身就太重，而且现有图谱更像展示图，不像真正可操作的结构工具。

## 9.2 改造方向
把它从“图谱”改成更易懂的“方向结构确认”。

### 面向普通用户的名字
- 不再叫“领域图谱”
- 改叫：
  - 方向结构
  - 内容方向
  - 生成结构确认

### 默认视图
默认用：
- 分组列表
- 层级卡片
- 可折叠树

而不是一上来强推图形布局。

### 图形视图
作为高级查看方式存在，不作为主视图。

### 用户动作
普通用户只做：
- 改名
- 删除
- 合并
- 补充
- 确认

不强迫用户理解边、节点、关系类型这些概念。

---

## 10. 错误与恢复机制重构方案

## 10.1 当前问题
错误几乎都是 Toast，一闪而过。

## 10.2 新错误系统要求

### 错误分级
- 轻微提醒
- 当前步骤失败
- 任务阻塞
- 系统异常

### 每类错误都要给
- 原因
- 影响范围
- 用户下一步
- 是否可重试

### 典型文案范式
不要写：
- 请求失败
- 任务异常

要写：
- 方向结构生成失败，原因是当前配置不可用。你可以：重试 / 更换模板 / 联系管理员。
- 导出未完成，因为仍有 12 条结果未生成完成。你可以：继续等待 / 仅导出已完成部分。

## 10.3 必须新增
- 持久错误面板
- 任务失败详情页
- 一键重试
- 从失败点恢复

---

## 11. 产品信任感重构方案

## 11.1 必须移除
- 登录页研发迁移信息
- 仓库路径
- 提交时间
- 开发环境口吻
- 默认账号直接暴露给终端用户

## 11.2 必须新增
- 清晰的品牌叙事
- 明确的任务完成通知
- 清晰的成功定义
- 导出成功后的确认与说明
- 对高成本动作的事前提示

## 11.3 高风险动作必须补齐说明
例如：
- 开始正式生成
- 覆盖已有结果
- 导出全部内容
- 删除任务

每个动作都要告诉用户：
- 会做什么
- 可能花多久
- 是否可撤销

---

## 12. 前后端接口重构要求

## 12.1 用户接口不再以“阶段按钮”为中心
不要继续围绕：
- `/questions/generate`
- `/reasoning/generate`
- `/rewards/generate`
- `/export`

让普通用户自己点完整流水线。

## 12.2 新接口方向

### 面向用户
- `POST /api/v1/tasks`
- `GET /api/v1/tasks/:id`
- `GET /api/v1/tasks/:id/progress`
- `POST /api/v1/tasks/:id/confirm-structure`
- `POST /api/v1/tasks/:id/retry`
- `GET /api/v1/tasks/:id/results`
- `POST /api/v1/tasks/:id/export`

### 面向管理员
保留现有：
- provider
- storage
- strategy
- prompt
- audit

但全部收敛到系统管理区，不再泄露到普通用户主流程。

## 12.3 后端必须补的能力
- 任务主记录表
- 阶段进度表
- 失败原因持久化
- 可重试状态
- 预计耗时估算
- 用户视角结果摘要接口

## 13. 页面级改版清单

## Phase A：先把产品“像个 TO C 产品”
- [ ] 重写登录页文案与布局
- [ ] 删除研发背景暴露信息
- [ ] 首页改成任务指挥页
- [ ] 新建任务页改成最小输入模式
- [ ] 普通用户隐藏 provider / storage / strategy 复杂项

**交付标准**：首次进入产品的普通用户，不需要理解内部系统结构，也能知道怎么开始。

## Phase B：建立真正的任务系统
- [ ] 引入 task_id 与任务状态机
- [ ] 引入任务详情页
- [ ] 引入阶段进度与 ETA
- [ ] 引入失败原因与重试入口
- [ ] 引入完成通知

**交付标准**：用户点击开始后，不再处于“已入队但不知道发生了什么”的状态。

## Phase C：重做结果中心
- [ ] 去掉内部字段直出
- [ ] 增加结果摘要卡
- [ ] 增加质量标签与异常说明
- [ ] 增加推荐下一步
- [ ] 增加导出说明与筛选导出

**交付标准**：结果页必须让普通用户看懂“产出了什么、质量如何、下一步做什么”。

## Phase D：重做方向结构确认页
- [ ] 把“领域图谱”改名为“方向结构确认”
- [ ] 默认改为树/分组视图
- [ ] 图形视图降级为高级模式
- [ ] 增加批量确认/批量调整
- [ ] 增加结构异常提示

**交付标准**：用户不需要懂图谱，也能完成结构审核。

## Phase E：错误恢复与信任建设
- [ ] 替换 Toast-only 失败体验
- [ ] 增加持久错误卡片
- [ ] 增加高风险动作说明
- [ ] 增加会话过期/权限失败的明确反馈
- [ ] 增加帮助中心与术语解释

**交付标准**：失败时用户知道为什么、怎么办、是否还能继续。

## 14. 设计原则（后续所有实现必须遵守）

1. **先让用户完成目标，再暴露系统能力**
2. **默认路径必须最短**
3. **用户不需要理解内部流水线**
4. **任何异步动作都必须有可见状态**
5. **任何失败都必须有恢复路径**
6. **任何高成本动作都必须有后果说明**
7. **用户页面禁止默认暴露内部实现字段**
8. **管理员复杂度不能转嫁给普通用户**

---

## 15. 最终验收标准

这次改版完成后，必须满足下面这些标准：

- [ ] 普通用户首次进入时，不会误以为自己进了后台
- [ ] 普通用户可以在 1 分钟内完成任务创建
- [ ] 普通用户不需要理解 provider / storage / graph / reward 等内部概念
- [ ] 任一生成任务都能看到明确阶段、进度、失败原因和重试入口
- [ ] 结果页不再像数据库记录页，而像成品内容页
- [ ] 高风险操作都有清晰后果说明
- [ ] 管理功能与普通用户主流程明确隔离
- [ ] 整体产品叙事从“数据工厂后台”变成“可直接使用的生成产品”

---

## 16. 一句话结论

本次改版不是“优化一下 UI”，而是：

> **把一个内部流水线控制台，改造成一个真正能打 TO C 市场的产品。**
